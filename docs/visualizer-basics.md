# 描画の基礎知識（ビジュアライザ制作のための）

このドキュメントは **「画面に描く側」の入門**です。音の解析は [audio-basics.md](audio-basics.md) と [features.md](features.md) に任せ、ここでは
**シェーダ / WebGL / GLSL / three.js / PixiJS / パーティクル / ポストプロセス** といった、
Web開発者にはなじみが薄い**CG・ゲーム畑の語彙**を、プログラマ向けの比喩でまとめます。

> 想定読者：JS・プログラミングは分かるが、GPU描画やシェーダは触ったことがない人。
> このプロジェクトでの対応は最後の章（[16](#16-このプロジェクトでの対応関係)）にまとめています。

---

## 1. 2つの描画方法：Canvas 2D と WebGL

ブラウザで絵を描く道具は大きく2つ。

- **Canvas 2D**（`ctx.fillRect(...)` の世界）：**手続き的に図形を置く**API。CPUで動く。素直で書きやすい。棒・円・線・テキストなど「普通のグラフィック」に最適。
- **WebGL**：**GPUに直接命令する**API。速いが低レベルで冗長。3D・大量の粒子・ピクセル単位のエフェクトはこちら。

このプロジェクトのプラグインも2種類あり、ちょうどこの2つに対応します（[Visualizer.ts](../src/visualizers/Visualizer.ts)）：
`Visualizer`（Canvas2D・手軽）と `SurfaceVisualizer`（自前のWebGL面を持つ・リッチ）。

---

## 2. CPU と GPU の違い（なぜGPUは速いのか）

- **CPU**：少数（数個〜数十）の賢いコア。複雑な分岐・逐次処理が得意。JSの `for` ループはこれ。
- **GPU**：**数千の非力なコア**。「同じ計算を大量のデータに一斉に適用する」のが得意（SIMD/並列）。

**比喩**：100万ピクセルそれぞれに同じ関数を適用するとき、CPUは `array.map()` を1コアで回す。GPUは**100万個をほぼ同時に**処理する。
だから「全ピクセルを毎フレーム塗り替える」ような処理はGPU（WebGL）が桁違いに速い。

---

## 3. WebGL とは

**ブラウザからGPUを叩くための低レベルAPI**（実体は OpenGL ES というモバイル向けGL規格のブラウザ版）。
できることは強力だが、三角形1枚出すのにもバッファ確保・シェーダのコンパイル・状態設定…と**記述量が多い**。
そのため実務では **three.js / PixiJS などのライブラリ**（WebGLを包んで使いやすくしたもの）を挟むのが普通です。

---

## 4. シェーダとは

**GPU上で動く小さなプログラム**。CPU（JS）ではなくGPUで実行される。

**比喩**：シェーダは「**頂点ごと／ピクセルごとに並列で呼ばれる純粋関数**」。
入力（座標や渡した値）を受け取り、出力（位置や色）を返す。副作用は基本なし。この関数がGPUの数千コアで一斉に走る。

シェーダは専用言語 **GLSL** で書きます（次章）。

---

## 5. GLSL（シェーダを書く言語）

**C風の、シェーダ専用言語**（GLSL ES）。JSではありません。特徴：

- 型が厳格：`float` / `int` / `bool` に加え、**ベクトル型 `vec2` `vec3` `vec4`**、**行列 `mat3` `mat4`**、テクスチャ `sampler2D`。
- 例：`vec3 color = vec3(1.0, 0.0, 0.0);`（赤）。`vec4`（RGBA）で色を返す。
- ループや関数は書けるが、**DOMも配列も文字列もない**。数値計算に特化。
- GPUで走るので、JSの変数を直接は読めない。**値は「uniform」経由で渡す**（[8章](#8-シェーダにデータを渡すuniform--attribute--varying)）。

---

## 6. 2種類のシェーダ：頂点シェーダ と フラグメントシェーダ

WebGLの描画は、必ずこの2つのシェーダのペアで動きます。

- **頂点シェーダ（vertex shader）**：**頂点ごと**に呼ばれ、その頂点の**画面上の位置**を決める（`gl_Position`）。形やカメラの計算。
- **フラグメントシェーダ（fragment shader）**：**ピクセル（フラグメント）ごと**に呼ばれ、その**色**を決める（`gl_FragColor` / 出力変数）。見た目の大半はここ。

「シェーダ芸」と言われるものは、たいていフラグメントシェーダで**1ピクセルずつ色を計算**して作ります。

---

## 7. 描画パイプライン（頂点 → ラスタライズ → ピクセル）

ざっくりの流れ：

```
頂点データ → [頂点シェーダ] → 画面上の三角形 → ラスタライズ（三角形を塗るピクセルに分解）
                                                   → [フラグメントシェーダ] → 各ピクセルの色 → 画面
```

- **ラスタライズ**：三角形の内部を「どのピクセルが含まれるか」に変換する工程（GPUが自動でやる）。
- 開発者が書くのは主に**両端のシェーダ**（頂点の位置と、ピクセルの色）。

---

## 8. シェーダにデータを渡す：uniform / attribute / varying

JSの値（例：`features.bass`）をシェーダで使うための3つの入口。**ここが音反応の肝**です。

- **uniform**：**1回の描画中は全頂点・全ピクセルで同じ値**。JSから毎フレーム設定する「グローバル定数」。
  → **`AudioFeatures` を渡すのはこれ**。例：毎フレーム `uBass = features.bass` をセットし、シェーダ内で色や動きに使う。
- **attribute**（頂点ごとの入力。WebGL2では `in`）：頂点座標・法線・UVなど、**頂点ごとに違う値**。
- **varying**（頂点→フラグメントの橋渡し。WebGL2では `out`/`in`）：頂点シェーダの出力を、ピクセル間で**自動補間**してフラグメントシェーダへ渡す。

**比喩**：uniform＝関数に渡す設定オブジェクト（全ピクセル共通）／attribute＝各要素固有の引数／varying＝要素間を滑らかに繋ぐ中間値。

---

## 9. フラグメントシェーダ一枚芸（Shadertoy / ISF スタイル）

画面全体を覆う**1枚の四角形**を置き、そのフラグメントシェーダで**ピクセルごとに色を計算**するだけで、
トンネル・プラズマ・波紋などの映像が作れます。頂点の形は使わず、**ほぼフラグメントシェーダだけで完結**するのが特徴。

よく使う入力（uniform）：
- **UV座標**（`0.0〜1.0` の画面内位置。左下(0,0)〜右上(1,1)）
- **時間**（アニメーション用。`uTime`）
- **解像度**（アスペクト比補正用）
- ここに **`uBass` / `uBeat` / `uTonalAngle`** などを足せば、そのまま音に反応する。

**Shadertoy** はこの形式のシェーダを共有するサイト、**ISF** は同種の交換フォーマット。競合 ShaderAmp もこの「シェーダ一枚芸」路線です。

---

## 10. メッシュ・ジオメトリ・マテリアル・テクスチャ（3Dの部品）

3D（three.js等）の基本語彙。

- **ジオメトリ（geometry）**：**形**。頂点の集まり（立方体・球・自作の頂点群）。
- **マテリアル（material）**：**表面の見た目**。色・光沢・透明度。**独自シェーダ（GLSL）を差し込める**のもここ。
- **メッシュ（mesh）**：**ジオメトリ ＋ マテリアル** ＝ 実際に置ける「物体」。
- **テクスチャ（texture）**：表面に貼る**画像**。フラグメントシェーダが `sampler2D` で色を読み取る。

**比喩**：ジオメトリ＝骨格、マテリアル＝肌、メッシュ＝完成した体、テクスチャ＝服の柄。

---

## 11. シーン・カメラ・レンダラ・ライト（3Dの舞台）

three.js のような**シーングラフ**型ライブラリの構成要素。

- **シーン（scene）**：**世界**。メッシュやライトを入れる入れ物（ツリー構造）。
- **カメラ（camera）**：**視点**。どこから・どんな画角で見るか（パース／並行投影）。
- **レンダラ（renderer）**：シーンをカメラ視点で**実際に描く**エンジン（`WebGLRenderer`）。
- **ライト（light）**：**光源**。3Dは光が当たって初めて立体に見える（環境光・平行光など）。

毎フレーム「シーンの中身を更新 → `renderer.render(scene, camera)`」を繰り返すのが基本ループ。

---

## 12. ライブラリの棲み分け：three.js / PixiJS / 生WebGL(GLSL)

| | 何に強いか | 作風 |
|---|---|---|
| **three.js** | 3Dシーングラフ（メッシュ・カメラ・光・ポストプロセス） | 立体・空間・カメラワーク |
| **PixiJS** | **GPU上の2D**（大量スプライト・パーティクル・フィルタ） | 2D映え・粒子・歪み/グロー |
| **生WebGL＋GLSL** | フラグメントシェーダ一枚芸 | シェーダアート（[9章](#9-フラグメントシェーダ一枚芸-shadertoy--isf-スタイル)） |

どれも下地はWebGL。**作りたい絵の系統で選ぶ**（3Dなら three、2D粒子なら Pixi、シェーダ純度なら生GLSL）。

---

## 13. ポストプロセス と パーティクル

- **ポストプロセス（post-processing）**：一度描いた画面を**テクスチャに焼いてから、全画面シェーダで加工**する手法。
  例：**ブルーム（bloom）**＝明るい部分をぼかして足す“光の滲み”。three.js の EffectComposer 等。
- **パーティクル（particles）**：**大量の小さな点/スプライト**を動かす表現（火花・星屑・煙）。
  GPUで数千〜数万を捌く。PixiJS の ParticleContainer、three.js の Points など。音の `kick`/`flux` で吹き上げる、といった用途に。

---

## 14. 座標系（NDC / UV / 画面ピクセル）

混乱しやすい3つの座標。

- **NDC（正規化デバイス座標）**：頂点シェーダが最終的に出す空間。画面の左下 `(-1,-1)` 〜 右上 `(1,1)`。中心が原点。
- **UV座標**：テクスチャ／画面内の位置を `0.0〜1.0` で表す。フラグメントシェーダでよく使う。
- **画面ピクセル座標**：`0〜width`, `0〜height`（Canvas2Dでおなじみ。y は下向き）。

**注意**：Canvas2D は「左上原点・y下向き・ピクセル単位」、WebGLは「中心原点・y上向き・-1〜1」。系が違うので変換が要る。

---

## 15. フレームループと FPS

- 画面は毎秒約60回描き替わる（**60fps**）。1フレームの持ち時間は **約16.7ミリ秒**。
- ブラウザでは **`requestAnimationFrame(loop)`** でこのタイミングに乗る（Web開発者にはおなじみ）。
- **16msを超えると“重い＝カクつく”**。重い処理はGPU（シェーダ）に逃がすのが定石。
- このプロジェクトでは、ホストがループを回し、プラグインの `draw()`（2D）や `frame()`（Surface）を**毎フレーム1回**呼びます。プラグイン側は「今フレームぶんの絵」を描くだけでよい。

## 16. このプロジェクトでの対応関係

用語が、実際のプラグイン契約（[Visualizer.ts](../src/visualizers/Visualizer.ts)）にどう対応するか。

- **`Visualizer`（2D・手軽）**：Canvas2D。`draw(features, { ctx, width, height })` に「今フレームの絵」を `ctx` で描くだけ。シェーダ知識ゼロでOK。
- **`SurfaceVisualizer`（自前描画面・リッチ）**：`mount(container)` で**自分のWebGLキャンバス/レンダラを作り**、`frame(features)` で毎フレーム描画、`unmount()` で片付け。three.js・PixiJS・生GLSL はここで使う。
- **リッチ描画の2経路**：
  - **内蔵プラグイン**（`src/visualizers/`）は `import * as PIXI from 'pixi.js'` のように**直接importしてバンドル同梱**（拡張・スタンドアロン両方に載る）。例：[PixiNeonVisualizer.ts](../src/visualizers/PixiNeonVisualizer.ts)（pixi-filters のフィルタも使用）。
  - **スタンドアロンのフォルダプラグイン**は、ホストが `window.MV.THREE` / `window.MV.PIXI` で提供するので同梱不要。例：[three-orb.js](../examples/plugins/three-orb.js)。
  - **GLSL一枚芸**は生WebGL＝ライブラリ不要（バンドルほぼ増えない）。[shaderSurface.ts](../src/visualizers/shaderSurface.ts) を土台に**フラグメントシェーダ本体だけ**書けばよい。例：[ChromaFlowVisualizer.ts](../src/visualizers/ChromaFlowVisualizer.ts)。
- **音を絵に繋ぐ = uniform**：`AudioFeatures`（`bass` `beat` `tonalAngle` …、[features.md](features.md)）を、シェーダなら **uniform** として毎フレーム渡し、3D/2Dなら位置・色・スケールに掛ける。**「解析済みの音データ × GPU描画」がこのプロジェクトの強み**。

イメージ（フラグメントシェーダに音を渡す例・擬似コード）:

```
// JS（frame ごと）
material.uniforms.uBass.value = features.bass;
material.uniforms.uBeat.value = features.beat ? 1.0 : 0.0;
material.uniforms.uTime.value = features.time;

// GLSL（フラグメントシェーダ）
uniform float uBass;   // 0〜1（低音の強さ）
uniform float uBeat;   // 拍の瞬間だけ1
uniform float uTime;
void main() {
  float glow = uBass + uBeat;          // 低音と拍で明るく
  gl_FragColor = vec4(vec3(glow), 1.0);
}
```
